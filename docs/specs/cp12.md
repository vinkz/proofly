## 1. Data Schema & Logic Mapping

The CP12 flow is split into **Record-level** data (one per job) and **Appliance-level** data (many per job). The wizard now follows the PDF sections directly and removes billable-customer capture for CP12.

---

### Record-Level Fields (`job_fields`)

These fields map to the **header and footer** of `cp12-template.pdf`.

#### Section 1 — Installer / Company (prefilled, not collected in wizard)
- Engineer name
- Company name
- Company address + Post Code
- Company Tel. No.
- Gas Safe registration number
- Engineer ID card number

> Source of truth: user profile. The wizard never asks for these; issuing is blocked until profile is complete.

#### Section 2 — Job Address (site)
- Address name / reference
- Address line 1, line 2, City/Town
- Post Code
- Tel. No. (site)

#### Section 4 — Customer / Landlord
- Name
- Company (optional)
- Address
- Post Code
- Tel. No.

#### Whole-House Checks (unchanged, future UI)
- `gas_tightness_satisfactory`
- `emergency_control_accessible`
- `equipotential_bonding_satisfactory`
- `pipework_visual_satisfactory`

#### CO Alarms (unchanged, future UI)
- Alarm fitted (yes/no)
- Tested and satisfactory (yes/no)

---

### Appliance-Level Fields (`cp12_appliances`)

Each appliance maps to **one row in the CP12 PDF table**.

#### Identity (Section 5)
- Location
- Appliance type
- Make
- Model
- Flue type

#### Inspection & Combustion (Section 6)
- Operating Pressure (mbar)
- Heat Input (kW/h)
- Combustion readings: High & Low (ratio/CO/CO₂) — stored as hi/lo ppm + CO₂ % equivalents
- Safety device(s) correct operation
- Ventilation provision satisfactory
- Visual condition of flue & termination satisfactory
- Flue performance test
- Appliance serviced
- Appliance safe to use

Limit: **5 appliances** (matches cp12-template table rows; additional appliances require a new certificate/job).

## 2. Validation & Business Logic

These rules enforce the CP12 specification and prevent invalid certificates from being issued.

---

### The “Hard Block” (Issue PDF Only)

Drafts can be saved with incomplete data.  
The **Issue PDF** action is blocked unless **all** of the following are true:

#### Mandatory Identity
- `property_address` is present
- `landlord_name` is present
- `landlord_address` is present

#### Minimum Content
- At least **one appliance** exists
- Appliance has both:
  - Location
  - Description

#### Legal Confirmation
- `reg_26_9_confirmed` (engineer declaration) is checked

#### Signatures
- Engineer signature captured
- Customer signature captured  
  *(both stored as image blobs)*

---

### Defect Logic

If **any appliance** is marked as **Unsafe** or a **defect** is recorded:

- `defect_description` becomes required
- `remedial_action` becomes required
- Engineer must specify whether:
  - **Labels and warning notices** were issued

These requirements are enforced **before PDF issuance**, not during drafting.
