# Repository Guidelines

## Project Structure & Module Organization
- `src/app/`: Next.js app router entry points, layouts, routes, and server actions.
- `src/components/` & `src/hooks/`: Reusable UI and client-side hooks; prefer colocated styles per component.
- `src/lib/`: Utilities, API clients, validation, and helpers shared across pages.
- `src/server/`: Server-only logic (Supabase helpers, PDFs, external APIs); keep fetchers and mutations here.
- `src/types/`: Shared TypeScript types and schemas.
- `public/`: Static assets served at the root.
- `tests/`: Vitest suites (e.g., `tests/server-actions.test.ts`) covering server actions and utilities.
- `supabase/` and `scripts/`: Supabase config and DevOps helpers (includes a legacy mobile tunnel script if ever needed).

## Agent Responsibilities
- AuthAgent: Handles login (password + magic link), password reset, password change, and signup wizard. Routes: `/login`, `/forgot-password`, `/reset-password`, `/signup/step1-3`. Server actions: `signInWithPassword`, `signInWithMagicLink`, `userHasPassword`, `changePassword`, `requestPasswordReset`, `applyPasswordReset`, `completeSignupWizard`.
- SignupAgent: Owns multi-step signup under `src/app/(auth)/signup` (steps: account details, personal/professional details, trade/cert selection) using shadcn UI and zod validation; finalizes via `completeSignupWizard` (auth signUp + profile upsert, onboarding_complete=true) then redirects to `/dashboard`.
- TemplateAgent: Filters templates by `profiles.trade_types` or `is_general`, keeps template metadata (`is_general`, `trade_type`) in sync, and wires template pickers/lists to those rules.
- ProfileAgent: Maintains `profiles` table fields (`trade_types`, `certifications`, `onboarding_complete`, personal basics, company details, Gas Safe IDs, request link slug, invoice defaults, `saved_signature_url`) via `src/server/profile.ts`, enforces guards in `RequireAuth` (redirects incomplete users to `/onboarding?step=N` based on missing profile fields, while allowing `/onboarding` and `/settings`). The guard must not self-redirect `/onboarding?step=N`; if the current pathname cannot be resolved from middleware/Next headers, require auth but skip the profile-completeness redirect. Profile setup must preserve Gas Safe ID rules: 6-digit business registration number and 7-digit individual ID card number where validation is enforced. `profiles.request_link_slug` is generated during signup and on demand by `getOrCreateEngineerRequestLink()` for dashboard empty states and `/jobs/new` landlord-detail prompts; prefer short readable slugs (`cn-[gas_safe_number]` where available, otherwise a short random/readable slug) and replace legacy long generated `cn-[uuid]` slugs opportunistically. Invoice pricing currently uses `profiles.standard_rates`, with a simple `default_rate` only as a possible compatibility/fallback field. `profiles.saved_signature_url` holds a 10-year signed URL to the engineer's canonical signature PNG in the `signatures` storage bucket (`signatures/{user_id}/profile-signature.png`); it is written by `saveProfileSignature(formData)` and cleared by `deleteProfileSignature()` in `src/server/profile.ts`. The `/settings` page (Milestone 4) has been fully redesigned using the new design system and is split into four separate cards: **Personal & engineer details** (name, DOB, profession, engineer name, Gas Safe reg, ID card number), **Company details** (company name, address, phone), **Invoices** (bank transfer fields + standard rates), and **Saved signature** (canvas capture via `SavedSignatureSection` at `src/app/(app)/settings/saved-signature-section.tsx`). A **Password** card and a **Subscription placeholder** card (pending Milestone 5 Stripe integration) follow below. The sticky save footer only covers the profile/company/invoice fields; signature changes are saved independently via their own action button.
- JobRequestAgent: Owns the bidirectional request/link lifecycle for landlord-submitted work requests and engineer-created jobs awaiting landlord data. It validates landlord request/pre-fill forms, classifies property/history-aware requests as `request_type = 'new_job'` or `request_type = 'renewal'`, creates/updates `job_requests` rows, triggers engineer/landlord notifications, and keeps request status moving through `pending -> scheduled -> completed -> dismissed`. It must support the three entry points: public landlord request with engineer lookup (`/request`, currently compatible with `/request-job`), engineer-created request-link prompt from `/jobs/new`, and engineer personal request link (`/request/[slug]` target). Matching should try `profiles.company_email`, Supabase Auth login email, Gas Safe number, and normalized phone formats before treating the engineer as unregistered. Existing matched engineers get an `Open prefilled job` email and must not receive create-account/claim copy; only unmatched engineers get the `Create account and claim request` email backed by `claim_token`/`claim_token_expires_at`. Engineer-only notifications must resolve to the matched engineer profile/Auth email where possible and must be skipped if that recipient equals the landlord email, so a landlord never receives the engineer CTA by field mix-up. If the landlord names an unregistered engineer, store the pending work on `job_requests` with the claim token as the canonical signup link rather than creating an ownerless job or relying only on email matching; the engineer may register with a different email. Public claim/pre-fill routes must validate tokens server-side through route handlers/server actions and must not add broad public RLS reads on `jobs`. Do not build a public browseable engineer directory.
- JobContextAgent: Produces the clean prefill payload for `/jobs/new` and downstream certificate wizards by merging context from the job request, scoped pre-fill form, property record, previous jobs/certificates, landlord/client record, and engineer profile. `/jobs/new` derives saved property options for existing clients from prior `jobs.client_id` + `job_fields` rows, including full Step 1 job address and landlord fields; selecting a saved property should prefill Step 1 and let the engineer continue straight into the certificate wizard from Step 2 where supported. When a saved property is chosen for a CP12, the new job may carry forward previous `cp12_appliances` identity fields only (type/location/make-model/flue type/landlord appliance flag) so appliance details are faster to re-enter without copying stale readings, safety classifications, warning flags, or live test results. It must preserve manual `/jobs/new` behavior when no request exists, and must never silently overwrite engineer-entered values in an active form.
- Boiler service saved-property creation follows the same identity-only rule: when `/jobs/new` starts a service from an existing saved property, copy prior boiler identity fields into `job_fields` (`boiler_make`, `boiler_model`, `boiler_type`, `boiler_location`, `serial_number`, `gas_type`, `mount_type`, `flue_type`) from the previous service job where available, falling back to CP12 appliance identity for make/model/type/location/flue type. Do not copy old combustion readings, operating pressure, heat input, safety answers, summary text, defects, signatures, or photos.
- JobLifecycleAgent: Owns the job-as-container state model and transition rules. Target job statuses are `draft -> awaiting_landlord -> prepared -> in_progress -> issued -> delivered`; these are additive to legacy statuses (`active`, `awaiting_signatures`, `awaiting_report`, `completed`) until dashboard/UI migration is complete. It determines required completion rows from `jobs.cert_types`, unsafe appliance flags in `job_fields`, and final `certificates` rows. `/jobs/[jobId]/complete` now exists as the authenticated checklist between certificate issue and delivery, with state assembled by `getJobCompletionState(jobId)` in `src/server/jobs.ts`; `/jobs/[jobId]/deliver` remains the next delivery/invoice-bundle phase. Minimum lifecycle columns were introduced in `supabase/migrations/20260512100000_minimum_job_lifecycle_fields.sql` and were reconciled on the remote Supabase project via MCP repair migrations.
- PropertyPageAgent: Owns the public-facing Property Vault at `/p/[public_token]` and the landlord-safe job view at `/j/[publicToken]` while the property route is introduced. It dynamically fetches all landlord-safe historical PDFs, active job/request statuses, invoice/payment status where available, renewal eligibility, engineer contact details, and service history tied to `property_id` or the best available legacy job-address grouping. It must never require login and must not expose internal UUIDs, private notes, service-role URLs, or unrelated customer history.
- AutofillAgent: Reduces repeated data entry across jobs, certificates, and invoices. It is the bridge between landlord-provided data and the engineer’s CP12/boiler-service wizard: data submitted through `/request-job`, `/p/[public_token]`, `/j/[publicToken]`, or an awaiting-input link should prefill Step 1 landlord/property/tenant/access fields while preserving engineer edits. It also autofills engineer company details, Gas Safe number, saved engineer signature where available, property address, landlord/client details, and eventually previous appliance data.
- CertificateAgent: Focuses on certificate validation, PDF generation, certificate storage, signed URLs, and certificate lifecycle side effects. It should not be responsible for gathering landlord/request data; it consumes normalized context prepared by JobContextAgent/AutofillAgent.
- InvoiceAgent: Owns invoice draft creation, line-item generation, totals, invoice PDF generation, payment-link metadata, and the finance side of the Handover Bundle. It calculates invoice totals from Standard Rates stored on `profiles.standard_rates` for now (`cp12`, `boiler_service`, `cp12_boiler_service`), then falls back to existing `invoices`/`invoice_line_items` last-used pricing, then zero/editable prices. The target state is zero-entry invoicing: certificate types on the job select the line items, standard rates supply prices, and the engineer only reviews before sending. `/jobs/[jobId]/deliver` cannot claim to attach invoices unless an invoice PDF renderer/storage path exists; build invoice PDF generation before or alongside delivery, or explicitly scope the first delivery version as certificate-only. MVP payment details are bank-transfer fields on `profiles`; PayPal/payment-provider links are future integration points stored through invoice payment-link metadata.
- DashboardAgent: Owns the operational dashboard request handoff and first-certificate activation. The top card remains the general welcome/action hero with `New Job` and `Create invoice`; the dashboard invoice CTA should guide engineers back to job-first invoicing until the delivery screen is complete. The next section is `Landlord Requests`, showing only the latest pending request plus a red badge count and a `/requests` link for the full list when requests exist. If there are no inbound requests, show a `Get started with CertNow` / `Create first certificate` card instead of prioritizing landlord-request sending. That card should show certificate-profile progress, prompt completion of certificate-required profile fields before certificate creation, show invoice/bank/rate details as optional setup, and end with a first CP12 certificate CTA. Keep the card title as `Create first certificate` even after jobs exist; once certificate profile and optional invoice setup are both complete, expose a local dismiss action so the engineer can hide the account setup card without a schema change. Dashboard also surfaces engineer-created `awaiting_landlord` shell jobs from `listAwaitingLandlordJobsForDashboard()` with the scoped prefill URL and a `Fill in myself` escape hatch; that action clears the token/awaiting state back to `draft` and opens the certificate wizard Step 1 without deleting the selected client/cert type. Dashboard rows now treat `issued` as a review/send state, showing `Review & send` and routing to `/jobs/[jobId]/complete`; legacy `completed` jobs still open the PDF view. Follow-up visit milestone counts must come from real gas-warning follow-up jobs (`certificate_type = 'gas_warning_notice'`, linked to a parent job, `draft`/`prepared`), not aged active jobs or seed data. Request visibility is milestone-critical because the new request flow is not useful if engineers cannot see inbound work from the dashboard.

## Architecture Overview
- Next.js App Router with mixed server/client components; pages and layouts live in `src/app`.
- Data flows through server actions in `src/app` and shared helpers in `src/server`; keep Supabase clients server-side except for SSR helpers in `src/lib`.
- Client UI pulls typed data via props; shared types in `src/types` keep server/client contracts aligned.
- Styling uses Tailwind (`tailwind.config.ts`); components favor utility-first classes over bespoke CSS.
- Production is intended to run on Vercel behind the custom domain `certnow.uk`; Vercel preview URLs remain useful for test deployments. Keep `NEXT_PUBLIC_SITE_URL` aligned with the exact public origin used for OAuth callbacks. For landlord/share links generated from local development, prefer `NEXT_PUBLIC_SHARE_URL=https://certnow.uk` so copied dashboard/request URLs are production-safe instead of `localhost`.
- Product direction is property-first and link-based: properties hold long-term compliance history, jobs are execution events at a property, and landlords can request work from public property links while engineers can still create jobs manually.
- Complete Job Lifecycle direction has three entry points into one shared job container: landlord uses a public request/lookup route, engineer creates a job manually or sends a landlord pre-fill link, and engineer shares a permanent personal request link. All entry points converge once the job is `prepared`, then produce a Handover Bundle containing certificate PDFs, invoice PDF, and payment link where available.
- Core model: Property = long-term compliance anchor; Job = one execution event at a property; Job Request = landlord-submitted request for work; Engineer Profile = reusable engineer/company identity for autofill.
- Every job must have a random, unguessable `jobs.public_token` generated at creation time. Public landlord/job routes must resolve by token, never by internal UUID.
- Target job lifecycle status is `draft -> awaiting_landlord -> prepared -> in_progress -> issued -> delivered`. Current status handling should be migrated additively; do not break existing dashboard prepare/start or certificate issue paths while introducing the new enum.
- App navigation remains engineer-first operationally: `/dashboard` is the operational home, `/jobs/new` must always support manual engineer-created jobs, `/properties` is the primary property history surface, `/clients` is the customer/contact history surface, and `/jobs` is available from the bottom nav in the slot formerly used by Docs. Job cards from `/jobs` and client history should route to `/jobs/[id]/complete` so engineers review the checklist before delivery rather than falling back into the old wizard/job-detail path.
- Document preview is canonical at `/jobs/[id]/pdf` (legacy report/pdf routes should redirect here); saved documents live under `/documents` and use Supabase signed URLs.
- External integrations: Supabase (auth/storage), PDF generation via `pdf-lib`, address lookup via Ideal Postcodes, transactional email via Resend, and error monitoring via Sentry; isolate integration code under server/API routes or framework instrumentation files. Sentry/Next instrumentation depends on OpenTelemetry hook packages being root-resolvable under Turbopack: keep `require-in-the-middle@8.0.1`, `import-in-the-middle@3.0.1`, and the pnpm override forcing `import-in-the-middle` to `3.0.1` unless Sentry/Next is intentionally upgraded and reverified.
- Job sheet scan/QR flows have been removed from the active product. Do not reintroduce QR scanning or drag-and-drop interactions without a clear mobile-field use case.
- Cross-certificate links: current CP12 can deep-link to Gas Warning Notice using the same jobId, and Gas Warning Notice may prefill from CP12 job fields/appliances. Target direction is to move this into `/jobs/[jobId]/complete`, where unsafe appliance flags make Gas Warning Notice a required checklist row and return the engineer to job context after issue.

## Business Model & Flywheel
- CertNow's paying customer is the engineer. Landlords are the distribution channel and retention/referral loop, not the direct customer.
- The core commercial offer is a solo-engineer subscription around `£12.99/month` for unlimited certificates, automated landlord reminders, permanent certificate storage, shareable job links, and renewal request notifications.
- Each completed job should automatically create landlord-facing value: a shareable property/job link proving compliance, certificate access, engineer contact details, reminder capture, and renewal request entry where appropriate.
- The intended flywheel is: engineer pays and uses CertNow on every job -> landlord receives a professional free link without login -> landlord perceives the engineer as organised -> landlord mentions it to other landlords -> those landlords ask their engineers to use CertNow -> more engineers subscribe.
- Engineer acquisition is the priority because engineers already pay for job-management tools, feel paperwork pain daily, decide quickly, and refer through trade networks. Landlord-facing features exist to make engineers stickier and more referable.
- Landlords should not be asked to pay in the core model. Their value is passive usage, trust, referral, and future compliance data, not direct SaaS revenue.
- Product decisions should favor anything that helps engineers finish work faster, look more professional, retain landlords, and trigger landlord word-of-mouth without adding CRM complexity.

## Validated Market Pain Points
- Missed CP12 reminders are usually accidental: emails go to spam, reminders are never sent, or landlords assume an agent handled it. CertNow schedules reminders as soon as the certificate is issued so renewal does not depend on memory.
- Possession and enforcement risk makes gas compliance high-stakes. Missing or invalid CP12 records can undermine a landlord's legal position, so CertNow should validate mandated fields, preserve timestamps, and keep an audit trail of documents and notifications.
- Renters' Rights Act context: GOV.UK landlord guidance says the 1 May 2026 changes removed the Section 21 no-fault eviction route and require landlords to use lawful possession grounds/processes. Treat this as product risk context, not legal advice; public copy should be reviewed before making legal claims.
- Lost certificates are treated operationally like missing certificates. Landlords often lose PDFs in email inboxes, WhatsApp threads, or local folders; CertNow should store certificates permanently against the property and expose them through stable shareable links.
- Certificate accuracy matters. Common omissions such as the landlord correspondence address can make a certificate unusable for compliance evidence, so CP12 validation should keep requiring legally mandated fields before issue.
- Access and tenant coordination create risk for landlords. Renewal request flows should capture tenant contact details and access notes so there is a clear record of attempts to arrange the visit.
- Portfolio tracking becomes difficult across CP12, EICR, and EPC dates. The long-term property dashboard should show red/amber/green status for all key property compliance documents and reduce reliance on spreadsheets.

## Competitor & Positioning Notes
- Gas Engineer Software and Gas Safety App are established engineer-facing competitors starting around `£7.50/month`. Treat them as validation that engineers pay for tools, but differentiate CertNow on mobile-first speed and landlord share links rather than desktop-style feature breadth.
- CertNudge is a landlord-facing CP12/EICR/EPC tracking competitor. Its weakness is manual upload: landlords must add documents themselves. CertNow's structural advantage is that CP12 certificates and reminders are created automatically because the engineer issues through CertNow.
- Initial market focus remains sole traders and micro-businesses. The strategy assumes most Gas Safe engineers are solo or very small teams, making fast individual purchasing and Facebook/trade-network referrals more practical than enterprise sales.
- Avoid building a public engineer directory by default. Landlords requesting work should be able to enter the engineer they already want to contact; open marketplace matching is a later business decision.

## EICR & EPC Tracking Scope
- EICR and EPC support should be document tracking only. CertNow must not generate EICR certificates because they belong to electricians and a different regulatory workflow.
- CertNow must not generate EPC certificates because EPCs require government accreditation and national register/Landmark integration.
- Correct scope: landlord or engineer uploads an existing EICR/EPC PDF against a property, confirms expiry date, and CertNow tracks compliance status and reminders using the same reminder infrastructure as CP12.
- Reminder direction: EICR reminders at six months before expiry; EPC reminders at twelve months before expiry.
- EICR/EPC tracking should appear on the property compliance dashboard alongside CP12 status. It is a planned expansion that makes the landlord view more valuable while preserving CP12 as the automatic engineer-led anchor.

## Targeting & Pricing Strategy
- Primary acquisition target: UK Gas Safe engineers, especially sole traders and micro-businesses, because they feel daily paperwork pain, already buy business software, and can decide in days to weeks.
- Primary engineer outreach channels: trade Facebook groups such as Gas Safe Engineers UK, UK Gas and Heating Engineers, The Gas Network, Worcester Bosch Installers Group, and direct outreach to small gas companies already promising annual reminders on their websites.
- Secondary distribution channel: landlords reached through engineer share links. Landlord community outreach should be helpful and informational only; do not post unprompted promotional content in landlord forums or groups.
- Planned pricing direction: solo engineer `£12.99/month`; company plan for two to five engineers around `£39/month` with admin dashboard and company branding; letting agent tier around `£49/month` for portfolio compliance; landlord accounts stay free and read-only.
- The free landlord compliance tracker for uploaded CP12/EICR/EPC documents is a later acquisition funnel, not a launch priority. It exists to create landlord referrals back to engineers.

## Property-First Job Model
- Property is the long-term compliance anchor and future Property Vault. It should own the persistent public landlord URL, compliance history, document vault, reminder eligibility, active job/request statuses, job request history, and future certificate history. Current job address/client structures must remain compatible while the property model is introduced.
- Job is one execution event at a property. Examples include CP12, boiler service, gas warning notice, general works, and commissioning. Jobs should be linkable to `property_id` and optionally to `job_requests.id` when created from a landlord request. Engineer-created jobs may exist before all landlord data is known; those jobs should retain a tokenized awaiting-input state rather than forcing the engineer to collect everything manually.
- Job Request is a landlord-triggered request to create work or a landlord-data collection state attached to engineer-created work. It is not itself a certificate; it becomes scheduled work only when the engineer accepts/schedules it into `/jobs/new`, or it enriches an existing job when the engineer initiated the workflow first.
- Engineer Profile is the persistent identity used to avoid repeated entry: company details, engineer name, Gas Safe number, ID card number, phone, default signature, and invoice/company defaults.
- Landlord/client records remain important for contact, billing, and portfolio context, but compliance status should be anchored to the property rather than duplicated across individual jobs.

## Complete Job Lifecycle Vision
- Bidirectional Entry: support both landlord-initiated inbound pull and engineer-initiated outbound push. Landlords can request or complete missing details from public links; engineers can always create jobs manually first and send a secure link when landlord details, tenant details, or access notes are missing.
- Property Vault: move the product away from a list of isolated jobs toward a property-centric vault. Each property should have one persistent public URL that exposes landlord-safe service history, active status, historical certificate PDFs, invoice/payment status where appropriate, and renewal/request actions.
- Handover Bundle: the final output of a job should be a combined compliance and finance package: CP12/boiler/GWN certificate PDFs, invoice PDF, and payment link when configured. Signature capture remains a prerequisite for issuing legally meaningful certificates and therefore for completing the Handover Bundle.
- Onboarding must be split into two gates: Certificate-ready is required before issuing PDFs; Invoice-ready is optional and only needed for automatic invoice/payment bundle completion. Missing invoice settings must never block certificate issue.
- This vision is additive. It must not break the existing CP12 wizard, `/jobs/new`, public job page, invoice routes, certificate PDF storage, or the current follow-up/reminder system.

## Job Creation Entry Points
- Entry Point 1, landlord public request: landlord starts from `/request` target route (current implementation compatibility: `/request-job`). The landlord enters the engineer contact they already know. This is not a public engineer directory. If an engineer profile is found, create a prepared job/request assigned to that engineer and notify them. If no profile is found, store `pending_engineer_email`/contact context and a `claim_token`, send an invite link such as `/signup?claim=...`, and attach the pending job to the new engineer account by token after signup rather than by email matching.
- Entry Point 2, engineer-created job: engineer starts from `/jobs/new` exactly as today. The start flow is staged as certificate type -> landlord/property -> start method. Landlord/property selection belongs before the choice between `Fill details myself` and `Ask landlord`, so both paths can use existing client/property context. Engineers can select an existing client and pick a saved property derived from prior `job_fields` to prefill Step 1; if the client has no saved property, they fall back to manual Step 1. Manual Step 1 save through `createSoloJob()` now creates a `prepared` job because the engineer has supplied the required landlord/property data. After selecting an existing client, the engineer can also call `requestLandlordJobPrefill()` to create an `awaiting_landlord` shell job with `prefill_token`/expiry and email `/prefill/[jobId]?token=...` to the landlord; for new landlords without a saved client, `/jobs/new` should use the engineer's personal `/request/[slug]` link email instead. The dashboard must show the waiting state and the `Fill in myself` escape hatch.
- Entry Point 3, engineer personal request link: each engineer should have a permanent `/request/[request_link_slug]` URL generated from their profile at signup and repaired on demand if missing or legacy-long. They can copy it from the dashboard empty request state, send it from `/jobs/new`, or share it through WhatsApp, business cards, or Google Business profile. Landlords using this link skip engineer lookup because the slug already scopes the request to the engineer. Submission creates a `job_requests` row assigned to the engineer immediately, sends the landlord only the landlord confirmation copy, and sends the `Open prefilled job` email only to the engineer account/profile email.
- Request type classification is separate from entry point. Once the property/history context is known, classify landlord-supplied demand as `request_type = 'new_job'` when no previous CertNow certificate history exists, or `request_type = 'renewal'` when previous CP12/certificate history exists and the certificate is due soon, within 60 days of expiry, or expired.
- All entry points converge once the job is `prepared`. Step 1 data is prefilled and editable; the engineer should enter CP12/boiler/GWN wizard work from Step 2 where practical. Manual engineer-created job = low/no prefill, landlord public request = medium prefill, renewal/property-linked request = high prefill.
- Engineer-created shell jobs in `awaiting_landlord` are active lifecycle rows, not just outbound emails. They retain `client_id`, `job_type`, `cert_types`, token expiry, and `landlord_input_requested_at`; `Fill in myself` clears token fields and returns to `draft`/manual Step 1 without losing the selected client/cert type.

## Job Lifecycle Status & Actions
- Target shared status model: `draft -> awaiting_landlord -> prepared -> in_progress -> issued -> delivered`.
- `draft`: engineer has started creating the job but Step 1 is incomplete.
- `awaiting_landlord`: shell job exists and a scoped pre-fill link has been sent to the landlord.
- `prepared`: Step 1 data is complete from engineer input or landlord pre-fill; engineer can start the wizard.
- `in_progress`: engineer has opened the wizard on site.
- `issued`: required certificate PDFs/signatures are generated but delivery is not complete.
- `delivered`: certificate/invoice bundle has been sent and the property record is updated.
- Target lifecycle actions should be split by ownership. `src/server/job-requests.ts` now owns `createPendingJobRequest`, `claimPendingJobRequest`, `submitStandaloneLandlordJobRequest` compatibility, and `sendJobRequestNotification`; it resolves existing engineers where possible and keeps unregistered-engineer work as claimable `job_requests`. `src/server/jobs.ts` now owns engineer-owned request/prep lifecycle actions: `requestLandlordJobPrefill`, `submitPrefillForm`, `fillAwaitingLandlordJobMyself`, `markJobPrepared`, and `getJobCompletionState`. Future work should add `markJobInProgress`, `markJobIssued`, and `createFollowUpJob` around existing `createJob`, `job_fields`, `job_requests`, and dashboard code rather than replacing current flows.
- Target delivery actions in `src/server/certificates.ts`: `buildDeliveryBundle(jobId)` and `sendDeliveryBundle(jobId, recipients, method)`. These should assemble existing `certificates` rows and invoice data; they should not move PDF generation out of the existing certificate renderers.

## Public Property Links
- Each property should have a public route `/p/[public_token]`. This route must never require login.
- The page is the Property Vault: it should show property address, current compliance/certificate status, latest certificate expiry date if available, engineer name and Gas Safe number, active job/request status, service history, and certificate/invoice download buttons if those documents are available for public sharing.
- CTA logic: if there is no previous CP12/certificate history, show `Request Gas Safety Check`; if there is previous CP12/certificate history and the certificate is due or expired, show `Request Renewal`.
- Both CTAs use the same small landlord form. The form must have four fields maximum: `tenant_name`, `tenant_phone`, `access_notes`, and `preferred_dates`.
- Do not ask landlords to re-enter property address, landlord name, landlord contact, or engineer details on the public form. Those should come from property/client/engineer records where possible.
- Public tokens must be random, unguessable, and scoped to the property. Public routes should expose only landlord-safe property/certificate data and must not expose internal UUIDs, private notes, service-role URLs, or unrelated client history.

## Public Request & Prefill Routes
- Current public standalone request route is `/request-job`; target route shape is `/request`. Keep `/request-job` working or redirect it when `/request` is introduced.
- `src/app/request-job/request-job-client.tsx` is a **3-step wizard** (May 2026 overhaul). All fields are React state — no FormData. `/request`, `/request-job`, and scoped `/request/[slug]` all keep the same step sequence: Step 1 = Engineer details, Step 2 = Your details/landlord details, Step 3 = Property + job address details. On scoped engineer links, Step 1 shows the resolved engineer read-only instead of asking the landlord to re-enter engineer details. Property/job step collects: tenant name (first field, replaces the old property reference field), address line 1 with Ideal Postcodes autocomplete, address line 2, city, postcode, "Same as your address" inline text button, tenant/site phone, access notes, job type chip group (`cp12`/`service`/`both`/`other`), and preferred date. `propertyReference` state and its `composeAddress` slot were removed. Job type uses a chip group, not a `<Select>`. Navigation uses Back/Continue/Send buttons outside the card; a progress bar and "Step N of M" indicator sit above the card.
- Target `/request` handles landlord-initiated requests where the landlord enters the engineer email/phone/Gas Safe reference they already have. It may create an assigned request/job, or an unclaimed pending job with `claim_token` if the engineer is not registered. The success state includes a WhatsApp share option for the engineer action/claim link when the landlord only has a phone number.
- Target `/request/[slug]` is the engineer personal request link. It skips engineer lookup and writes the request/job against the engineer resolved from `profiles.request_link_slug`; this is the preferred link sent from `/jobs/new` and shown in the dashboard empty request state. Landlord-facing copy should stay short and practical: `Book a gas engineer`, dynamic engineer name where available, and `No account needed.`
- Target `/prefill/[jobId]?token=...` is for engineer-created shell jobs in `awaiting_landlord`. It must be token-scoped to one job, public/no-login, and write only allowed Step 1 pre-fill fields before setting the job to `prepared`.
- Public pre-fill forms should stay one-screen and mobile-first: property address fields, tenant name/contact, cert type (`cp12`, `boiler_service`, or both), and preferred visit date. Use the same Ideal Postcodes-backed address formatting as CP12/boiler Step 1.

## Job Completion & Delivery Routes
- `/jobs/[jobId]/complete` is the job-as-container checklist inserted between wizard signature completion and delivery. It is an authenticated app route, not a landlord/public route. It currently reviews required certificate/notices and optional invoice readiness, and keeps delivery/invoice bundle sending scoped to a later `/jobs/[jobId]/deliver` phase.
- `/jobs/[jobId]/complete` was implemented additively. Existing CP12/boiler/GWN PDF generation still works, but final generation now moves the job to `issued` instead of treating it as dashboard-completed. Legacy jobs with missing `cert_types` infer required rows from `jobs.job_type` and existing certificate rows where possible rather than crashing or hiding the job.
- Core server action: `getJobCompletionState(jobId)` in `src/server/jobs.ts`. It validates authenticated ownership, reads `jobs`, `job_fields`, `cp12_appliances`, `certificates`, and invoice state, then returns checklist rows split into required and optional states. Do not put PDF rendering in this action; it should only inspect existing job/certificate state and produce UI state.
- Required certificates come from `jobs.cert_types`: `cp12` requires a final CP12 certificate row; `boiler_service` requires a final boiler service certificate row. For backward compatibility, `job_type = 'safety_check'` implies CP12 and `job_type = 'service'` implies boiler service when `cert_types` is empty.
- Required notices come from unsafe appliance flags in `job_fields` (`ID` or `AR`) plus the required defect/remedial data. If an unsafe appliance exists and no final Gas Warning Notice certificate exists for the job, the completion screen shows `Gas warning notice — Required`.
- Missing required final certificates/notices block the Send CTA. The invoice row is `Ready` or `Draft needed`, but invoice setup must not block certificate issue; it only affects whether the handover bundle can include a complete invoice immediately.
- Boiler service and Gas Warning Notice opened from `/jobs/[jobId]/complete` should return the engineer to `/jobs/[jobId]/complete` after completion as routes are migrated. Thread `jobId` through the wizard as a route/search param and revalidate the completion route after storing the final certificate row.
- CP12 and boiler service final issue actions now redirect to `/jobs/[jobId]/complete`; do not add new post-certificate detours that bypass the completion checklist. Unsafe CP12 appliances make Gas Warning Notice a required row on the same job rather than creating separate draft warning-notice jobs at CP12 issue time.
- Gas Warning Notice issue is immutable for a job: once a final GWN certificate row with a PDF exists, the GWN wizard route must redirect back to `/jobs/[jobId]/complete` and the server action must refuse to overwrite it. Issuing a GWN creates one draft remedial follow-up job linked by `parent_job_id` and `job_fields.gas_warning_follow_up_source_job_id`, with the same client/property context and defect/remedial fields prefilled.
- Dashboard routing should support the completion screen: `issued` jobs show `Review & send` and route to `/jobs/[jobId]/complete`; `prepared` jobs continue to start the wizard; legacy `completed` jobs can keep opening the existing PDF/job pages until migration is complete.
- Target `/jobs/[jobId]/deliver` is reached only after required items are resolved. It handles recipient choice, send method, inline invoice editing, and sends all job documents as one Handover Bundle.
- Delivery/invoice sequencing rule: if `/deliver` includes invoice sending, implement `renderInvoice`/invoice PDF storage in the same phase so Resend can attach a real invoice PDF. If invoice rendering is not ready, the first `/deliver` version must be certificate-only, with invoice marked `Draft/Not included yet` and no misleading invoice attachment/status copy.
- Invoice PDF renderer should follow the existing server PDF pattern: generate from job/client/profile/invoice line data, store in Supabase storage alongside other job documents, link to `invoices.job_id`, and expose signed URLs only through authenticated app routes or landlord-safe public bundle routes.
- Current CP12 post-issue flow now offers the completion checklist after certificate issue. Avoid adding more one-off post-certificate prompts; route new post-issue requirements through `/jobs/[jobId]/complete`.

## Email Sending Policy
- CertNow owns the sending infrastructure for product emails. Transactional messages should send from a CertNow-controlled sender such as `notifications@certnow.uk` or `general@certnow.uk`, using Resend or the configured provider.
- Shared Resend delivery lives in `src/lib/resend.ts` via `sendEmail()` and `isEmailConfigured()`. Do not create feature-local `fetch('https://api.resend.com/emails')` helpers; `/request`, `/prefill`, reminders, delivery bundles, invoice sends, and claim invites should all use this utility.
- Emails should still feel like they come from the engineer relationship. Include the engineer/company name prominently in subject/body copy where relevant, and set `Reply-To` to the engineer/company email when available so landlord replies go to the engineer, not CertNow support. Keep landlord-facing request-link/confirmation copy separate from engineer-facing request notification/claim copy; do not send `Open prefilled job` or `Create account and claim request` CTAs to the landlord email.
- Do not send directly from arbitrary engineer email addresses in the MVP. Per-engineer sender domains create SPF/DKIM/domain-verification and deliverability support burden. Treat custom sender domains as a future paid/advanced feature.
- For landlord request, job completion, certificate, invoice, and reminder emails, CertNow is the system sender and the engineer is the represented service provider. Copy should say or imply `Sent on behalf of [Engineer / Company]` where that context matters.
- Email links should point to stable CertNow routes: `/jobs/new?requestId=...` for engineer request acceptance and `/j/[publicToken]` for landlord-safe job/certificate views. Avoid exposing internal UUIDs in public landlord links.

## Security Boundaries
- Any module that reads private environment variables (`OPENAI_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY`) must stay server-only and include `import 'server-only';` unless it is a top-level Next server action file with `'use server'`.
- Current guarded secret-bearing utilities: `src/lib/env.ts`, `src/lib/openai.ts`, `src/lib/reporting.ts`, `src/lib/supabaseServer.ts`, and `src/server/pdf/renderCp12Certificate.ts`.
- `src/lib/supabaseServer.ts` owns shared SSR Supabase auth helpers. `getSupabaseUser()` retries transient Supabase Auth `Request rate limit reached` errors before failing, and profile/onboarding server paths should use it instead of raw `auth.getUser()` when possible.
- Do not import private server utilities directly into client components. Client components may import server actions from `'use server'` modules, but those actions must validate the authenticated user and resource ownership before using service-role clients.
- `supabaseServerServiceRole()` bypasses RLS. Use it only in server actions, route handlers, or server-only helpers after confirming the current Supabase user is allowed to access the job/client/certificate being read or mutated. Service-role Supabase clients must be created with `createClient()` from `@supabase/supabase-js` and `persistSession: false`, not the SSR cookie client, so request cookies cannot cause service writes to inherit anon/authenticated RLS behavior. Vercel must have the real `SUPABASE_SERVICE_ROLE_KEY`, not the anon key, for public request server actions to write `job_requests`.
- Never expose, serialize, log, or return private keys or key-presence checks. `NEXT_PUBLIC_*` values are public and must not contain secrets.
- Before real customer rollout, audit RLS policies and storage bucket rules for at least `profiles`, `properties`, `job_requests`, `clients`, `jobs`, `job_fields`, `certificates`, `invoices`, `documents`, `reports`, and certificate storage buckets.
- Public property routes (`/p/[public_token]`) must be deliberately unauthenticated but data-minimized. Use the token to resolve only landlord-safe fields, never service-role URLs or unrelated customer/job history.
- Public claim/pre-fill routes must validate `job_requests.claim_token` or `jobs.prefill_token` on the server. Do not grant unauthenticated table-level reads on `jobs` just because a token exists; that would leak owner data and break the engineer-owned security model.

## Auth & Onboarding
- Auth supports Google OAuth, magic-link (`signInWithOtp`), password login, password change, and reset flows; environment requires Supabase keys and `NEXT_PUBLIC_SITE_URL` for OAuth/reset redirects.
- Google OAuth uses `src/components/auth/google-auth-button.tsx` and redirects through `/auth/callback`. The callback exchanges the Supabase code, checks onboarding/profile completeness, and sends complete users to `/dashboard`.
- For custom domains and mobile testing, Supabase Auth redirect URLs must include the exact callback origins in use, e.g. `https://certnow.uk/auth/callback`, `https://www.certnow.uk/auth/callback`, and any stable Vercel/staging callback URL. After changing `NEXT_PUBLIC_SITE_URL`, redeploy/restart because it is bundled into client code.
- Unified signup + onboarding is handled by the wizard at `/signup/step1-3`; incomplete profiles redirect there. Auth helpers include `userHasPassword`, `completeSignupWizard`, `changePassword`, `requestPasswordReset`, and `applyPasswordReset`.
- Signup creates the auth account at `/signup/step1`, then compatibility routes `/signup/step2` and `/signup/step3` send the user into the protected `/onboarding?step=N` wizard instead of looping backwards.
- Signup Step 1 is idempotent for testing existing email/password accounts: if Supabase reports the email is already registered, `signUpWithPassword` attempts a normal password sign-in and continues to `/signup/step2` when credentials are valid. It must not bypass password checks or silently take over a different user.
- `src/server/profile.ts` uses shared `getSupabaseUser()` so `/onboarding` and `RequireAuth` tolerate transient Supabase Auth rate limits instead of crashing the app shell.
- `RequireAuth` reads the current pathname from `x-current-path` set by `middleware.ts`, with fallbacks for Next request headers. Exact protected root paths such as `/onboarding`, `/dashboard`, `/jobs`, and `/settings` must remain in the middleware matcher as well as their `/:path*` variants. If pathname resolution fails, the guard should still require a Supabase user but must skip incomplete-profile redirects to avoid an `/onboarding?step=N` 307 loop.
- Onboarding step 1 collects personal basics, step 2 collects company details, and step 3 collects Gas Safe registration number plus engineer ID card number. These are persisted to `profiles`; `onboarding_complete` is true only when all required fields are present and the Gas Safe IDs match the 6-digit business / 7-digit individual format.
- If signup fails with `permission denied for table profiles`, fix the `public.handle_new_user()` trigger to be `SECURITY DEFINER` so it can insert into `public.profiles` under RLS.

## Wizard Flow Notes
- Job address steps follow the shared card layout: job reference, address name/lines/city/postcode, and site telephone; job line1/postcode sync into property address fields.
- Address lookup should continue to use the existing Ideal Postcodes-backed `/api/address-search` path for all job/property address entry points. Do not introduce a second address provider or inconsistent address formatting unless explicitly planned.
- `/jobs/new` is the convergence point for manual engineer-created jobs, landlord new-job requests, and landlord renewal requests. It should accept optional context from `job_requests.id`, property, client/landlord, prior certificates, and engineer profile without breaking the existing manual creation path.
- Job context prefill priority should be: explicit job request values, property record, previous jobs/certificates, landlord/client record, engineer profile defaults. Engineer-entered values in the current form should not be silently overwritten.
- General Works wizard step order: Job address → Evidence → Review → Signatures (client step still precedes wizard).
- Gas Warning Notice job step includes job address fields plus customer contact card; the job address fields are stored in job_fields and used to update the job address via `saveGasWarningJobInfo`.
- Boiler Service is a public/selectable engineer flow again. It should start directly in `/wizard/create/gas_service`/`/wizard/create/boiler_service` like CP12, not behind the old client-only pre-step. Step 1 mirrors CP12 where relevant: service date, structured job address, address API lookup, site telephone, and structured client/landlord correspondence details.
- Boiler Service visible fields should track the PDF template, not generic service notes. The UI must collect every gas-service PDF field that is not already autofilled from profile/job context: job/client address blocks, boiler identity, high/low combustion readings, operating pressure, heat input, PDF safety/template yes-no checks, summary/recommendations/comments, next service date, and engineer/customer signatures. Avoid adding visible fields that are stored but never render to the Boiler Service PDF unless there is a clear operational reason.
- Boiler Service and CP12 voice readings share `/api/cp12/voice-readings` and `src/lib/cp12/voice-readings.ts`. Keep voice capture as an inline input helper for visible fields, not a separate evidence workflow. Use one pressure/input action (`scope="pressure"`) for operating pressure + heat input. Combustion capture should default to one action per firing state (`scope="high"` for high-rate CO/CO2/ratio and `scope="low"` for low-rate CO/CO2/ratio), because high and low readings are normally captured while the appliance is in different operating states. The backend still supports `scope="combustion"` for future combined capture, but avoid a broad "Speak FGA readings" UI button unless the product deliberately changes to a combined high/low recording flow. FGA printout/photo evidence should stay optional and separate from spoken readings.
  - Scoped voice parsing supports ordered-number fallback. In pressure scope, bare numbers map to `operating pressure -> heat input`; in high/low scope, bare numbers map to `CO ppm -> CO2 % -> ratio`. Keep the UI copy short but make this review requirement clear because ordered parsing is convenient but can mis-fill if the engineer speaks values in the wrong order.
- Offline mode Level 1 is local draft capture only. `useWizardDraft` persists loaded CP12 and Boiler Service wizard state to durable browser storage, shows not-synced/offline status via `OfflineDraftBanner`, lets engineers move between draft steps while offline, and attempts to sync text/check/appliance data when the browser comes back online. Do not issue PDFs, send emails, mark jobs delivered, or claim uploaded photos/signatures are safely synced while offline; binary uploads still require online storage work.
- CP12 (2026-02 refresh):
  - Section order mirrors the PDF: Installer (read-only from account) → Job address → Customer/Landlord → Appliance identity → Appliance checks → Signatures.
  - **Billable customer is removed** from CP12; only job address + landlord/customer are collected.
  - Installer/company + Gas Safe + ID card come from profile; if missing, issuing is blocked and the user is pushed to Settings.
  - Step 1 now supports inline **client selection/creation** inside the wizard. New clients require only `name`; `phone` and `email` are optional. The job stores the linked `client_id` as the canonical key.
  - When a client is linked, Step 1 prefills from the canonical client record and can also offer **saved properties** derived from that client’s prior jobs/job fields.
  - Saved property selection fills property-side fields only: `job_address_name`, `job_address_line1`, `job_address_line2`, `job_address_city`, `job_postcode`, and `job_tel`. Landlord/property-owner fields remain separate for CP12 compliance.
  - Step 1 Job location requires `job_address_name`, `job_address_line1`, `job_postcode`, and `job_tel` (site telephone); `job_address_line2` and `job_address_city` are supported and render on separate PDF address lines.

For backward compatibility, `landlord_address` is still persisted and used as a fallback.

These persist to `job_fields` and update the linked client email/phone so future jobs for that client prefill landlord contact automatically. Missing landlord email shows a non-blocking advisory on the final step; it must never block CP12 issue.
  - CP12 Step 1 also supports a **prepare-only** entry path from dashboard upcoming jobs via `?prepare=1`; saving People & Location persists Step 1 data and returns to `/dashboard` without forcing the rest of the wizard.
  - Appliances capped at 5 rows to match the PDF table capacity.
  - Mobile UX is intentionally tight: demo-fill buttons are hidden, address lookup disabled/configuration errors are suppressed in the UI, Step 2 has no extra "Appliance profile" wrapper, and `+ Appliance` is inline with "Appliance 1 identity".
  - Step 3 no longer exposes a global "Measurement source" switch. Voice capture is scoped inline by workflow group, not by every individual field: `Speak pressure` fills operating pressure + heat input, `Speak high` fills high CO/CO2/ratio, and `Speak low` fills low CO/CO2/ratio. Scoped recordings may say bare high/low phrases like `CO 8, CO2 9.2, ratio 0.0008`; the parser maps them into the active group server-side via `/api/cp12/voice-readings` `scope`. The inline voice button shows subtle hint text in idle state and should make clear which visible field group is being recorded into.
  - Step 4 checklist uses SVG icon circles instead of emoji: green filled circle + checkmark for passing items, amber circle + exclamation for blocking items, gray circle + info for non-blocking items. A green "Ready to issue" banner appears inside the checklist card when `checklist.blockingMissing === 0`. The sticky footer bar has three buttons: Edit (flex-1 outline, goes to step 1), Send to landlord (flex-1 outline, calls `handleCreateRemoteSignatureLink`), Issue CP12 (flex-[2], `bg-[#1a7a52]`). - CP12 may still create/deep-link Gas Warning Notice work from unsafe appliance handling for compatibility, but the primary post-issue direction is the `/jobs/[jobId]/complete` checklist. Unsafe appliance flags should make Gas Warning Notice a required completion checklist item before delivery.
- CP12 guardrails (docs/specs/cp12.md): property + landlord addresses required and must differ, landlord name required, Reg 26(9) confirmation required, at least one appliance with location & description, engineer + customer signatures required to issue, and if any appliance is unsafe/defective you must capture defect_description, remedial_action, and a warning_notice_issued choice.

## Follow-Up Logic
- CP12 is the main compliance lifecycle anchor. When a CP12 is completed, always create a 12-month CP12 follow-up.
- CP12 issue also schedules reminder rows: landlord at eight weeks and four weeks before next inspection due, and engineer at eight weeks before. `/api/cron/reminders` processes due reminder rows daily and sends through the configured CertNow email provider. Only mark a reminder `sent_at` after a successful send; missing recipients or provider failures should remain visible in cron output rather than being silently discarded.
- Boiler Service follow-up depends on context. If linked to CP12 on the same job, do not create a separate boiler service follow-up because the CP12 renewal cycle covers it.
- If a Boiler Service is standalone, create a 12-month boiler service follow-up and add note: `Standalone service — confirm whether CP12 also required`.
- Gas Warning Notice keeps its existing logic: it is triggered from unsafe CP12 appliance checks and should not be merged with boiler service follow-up logic.
- Known future gap: standalone boiler service finding a dangerous appliance may need separate Gas Warning Notice handling later.

## Invoice Completion Flow
- Invoices already exist under `/invoices`, `/invoices/new`, and `/invoices/[invoiceId]`, but certificate completion should integrate them more directly.
- At CP12 Step 4 completion, after successful PDF generation, prompt: `Create invoice for this job?`
- CP12 completion first asks whether a boiler service was also completed. If yes, open `/wizard/create/boiler_service?jobId=...` on the same job so the boiler service certificate links to the CP12 job. If no, show invoice options and the `/jobs/[jobId]/complete` checklist CTA.
- The invoice CTA should open `/invoices/new?jobId=...` and create one draft invoice for the job with line items derived from all certificates on that job. Unit prices default to the engineer’s last used price for that certificate type when available; otherwise they remain zero/empty for editing.
- Engineers should not need to navigate separately to invoices after completing a certificate, though standalone invoice creation from the dashboard/header should remain available.
- Handover Bundle target: after certificate issue and required signatures, the job completion flow should assemble all available certificate PDFs, one invoice PDF, and a payment link/status into a single landlord-facing package on the public property/job link. Invoice totals should come from Standard Rates on the engineer business profile where available, falling back to current last-used price logic until the rate model exists. If the bundle is delivered before invoice PDF generation exists, call it certificate delivery only and keep invoice actions visible but separate.
- Current implementation direction: `profiles.standard_rates` stores CP12, Boiler Service, and CP12 + Boiler Service prices. Existing bank-transfer profile fields are the MVP payment method. Future PayPal support should plug into invoice `payment_link_url` / `payment_status` without making certificate issue depend on payment setup.
- Target delivery model: invoice editing moves inline to `/jobs/[jobId]/deliver`, with no separate navigation needed in the clean case. Standalone `/invoices` routes remain for manual invoice admin and backwards compatibility.

## Engineer Tools
- Utility tools should be reachable from a top-right app menu rather than the fixed bottom nav. Keep bottom nav focused on daily operational surfaces; use the top menu for secondary tools such as Gas Rate Calculator and standalone invoice admin.
- Gas Rate Calculator backend lives in `src/lib/gas-rate-calculator.ts`, `src/server/gas-rate.ts`, and `POST /api/tools/gas-rate`. The calculator accepts metric timed volume in m3 or imperial timed volume in ft3, optional start/end readings, test duration in seconds, editable calorific value, and editable correction factor. Defaults are `39.5 MJ/m3` and `1.02264`, but UI must let engineers override these because actual CV can vary by supplier/area.
- Gas rate output is advisory calculation support, not certificate issuance state. Do not persist calculator results to jobs unless the engineer explicitly copies/uses the value in a visible certificate field.
- Standalone invoice generation already exists at `/invoices/new` and should be linked from the same top-right tools menu. The menu CTA should route to `/invoices/new` or `/invoices/new?guided=1`; do not create a second invoice engine.

## Compatibility Guardrails
